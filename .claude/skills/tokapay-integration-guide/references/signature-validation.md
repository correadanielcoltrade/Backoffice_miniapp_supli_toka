# Signature Validation — Deep Dive

Canonical spec for how requests/responses between Toka and merchant systems are signed and verified. Applies to **all** Tokapay OpenAPIs — POS, Mini-Program, and Agreement Pay.

> **Source**: *Solución de Validación de Firmas* (Toka).

---

## 1. Key Pair Requirements

Signing is **mutual and asymmetric**. **Four** `.pem` files total — two pairs.

| # | Key file | Held by | Used to | Generated by |
|---|---|---|---|---|
| 1 | `ctoka_private_key.pem` | **Toka** (secret) | Sign Toka's requests/responses | Toka |
| 2 | `toka_public_key.pem` | **Merchant** | Verify Toka's signatures | Toka |
| 3 | `paypay_private_key.pem` | **Merchant** (secret) | Sign merchant's requests/responses | Merchant |
| 4 | `paypay_public_key.pem` | **Toka** | Verify merchant's signatures | Merchant |

> Simple rule: **the private key signs, the public key verifies.** Each party generates its own pair and ships only the public half.

Public key exchange is **offline** (email/shared drive) during onboarding. No online endpoint to fetch or rotate keys.

---

## 2. Generating Your Keys

```bash
openssl genrsa -out rsa_private.pem 2048
openssl rsa -in rsa_private.pem -pubout -out rsa_public.pem
```

- RSA length: 1024 or 2048 accepted. **Use 2048** for production.
- Keep the private key on the server; never commit it.
- Send the public key to Toka offline.

## 3. Message Structure

### Request
```
Method:      POST
URL:         https://{domain}/{version}/{restfulPath}
Headers:
  Signature:     algorithm=SHA256withRSA, keyVersion=1, signature=<base64UrlEncoded>
  Content-Type:  application/json;charset=UTF-8
  Client-Id:     <merchant client id>
  Request-Id:    <unique random string per request>
  Request-Time:  <millisecond unix timestamp, e.g. 1685599933871>
Body:        JSON
```

**Algorithm**: `SHA256withRSA` (default). `SHA512withRSA` also supported.
**keyVersion**: fixed to `1` until a rotation is announced.

### Response
```
Headers:
  Signature:     algorithm=SHA256withRSA, keyVersion=1, signature=<base64UrlEncoded>
  Client-Id:     <merchant client id>
  Response-Time: <millisecond unix timestamp>
Body:        JSON with a `result` object
```

> Header field names are **case-insensitive**.

## 4. Signing a Request (merchant → Toka)

### Step 1 — Construct `Content_To_Be_Signed`

**Six parts joined by dots** — not just the body:

```
<Http-Method>.<Http-Uri-With-Query-String>.<Client-Id>.<Request-Id>.<Request-Time>.<Http-Body>
```

| Part | Example |
|---|---|
| `Http-Method` | `POST` |
| `Http-Uri-With-Query-String` | `/v1/acquiring/payment/create` (path only — **strip the domain**) |
| `Client-Id` | `toka` |
| `Request-Id` | `gjt0d0l6uqn` (random per request) |
| `Request-Time` | `1685599933871` (**millisecond** precision) |
| `Http-Body` | The exact JSON you'll send — character-for-character |

Worked example:
```
POST./v1/acquiring/payment/create.toka.gjt0d0l6uqn.1685599933871.{"productCode":"QR_STATIC_PAYMENT","shopId":"15",...}
```

> The body in the signing string must be **byte-identical** to what's transmitted. Whitespace, key order, and escaping matter. **Sign the exact serialized string that goes on the wire.**

### Step 2 — Generate the signature
```
generatedSignature = base64UrlEncode( sha256withrsa(<Content_To_Be_Signed>, <privateKey>) )
```

### Step 3 — Build the header
```
Signature: algorithm=SHA256withRSA,keyVersion=1,signature=<generatedSignature>
```
Three comma-separated `key=value` pairs. No spaces around `=`.

### Step 4 — Send
```bash
curl -X POST https://openapi.paypay.mx/v1/acquiring/payment/create \
  -H 'Content-Type: application/json' \
  -H 'Client-Id: toka' \
  -H 'Request-Id: gjt0d0l6uqn' \
  -H 'Request-Time: 1685599933871' \
  -H 'Signature: algorithm=SHA256withRSA,keyVersion=1,signature=j1cnsY6...' \
  -d '{...}'
```

## 5. Verifying a Response (Toka → merchant)

### Step 1 — Construct `Content_To_Be_Validated`

**Only three parts** (no method, URI, or request-id):

```
<Client-Id>.<Response-Time>.<Response-Body>
```

> **This is the most common mistake.** Engineers reuse the 6-part request format and verification fails. Responses use **3 parts only**.

Example:
```
toka.1685599933871.{"result":{"resultStatus":"A","resultCode":"20000006",...},"paymentId":"..."}
```

### Step 2 — Extract the target signature
Parse the `Signature` header, pull the `signature=` value.

### Step 3 — Verify
```
IS_SIGNATURE_VALID = sha256withrsa_verify(
  base64UrlDecode(<target_signature>), <Content_To_Be_Validated>, <toka_public_key>
)
```

- `true` → body is authentic.
- `false` → keys don't match, or content was built wrong. **Reject the response.**

> **Edge case**: When Toka rejects the request with `SIGNATURE_INVALID`, the response may contain **no signature at all**. Treat a missing signature as a verification failure.

## 6. Reference Implementation (Java)

### Sign
```java
public static String sign(String requestURI, String clientId, String requestId,
                          String requestTime, String privateKey, String requestBody) {
    String content = String.format("POST.%s.%s.%s.%s.%s",
        requestURI, clientId, requestId, requestTime, requestBody);
    java.security.Signature signature = java.security.Signature.getInstance("SHA256withRSA");
    PrivateKey priKey = KeyFactory.getInstance("RSA").generatePrivate(
        new PKCS8EncodedKeySpec(Base64.decodeBase64(privateKey.getBytes("UTF-8"))));
    signature.initSign(priKey);
    signature.update(content.getBytes("UTF-8"));
    byte[] signed = signature.sign();
    return URLEncoder.encode(new String(Base64.encodeBase64(signed), "UTF-8"), "UTF-8");
}
```

### Verify
```java
public static boolean verify(String clientId, String responseTime,
                             String tokapayPublicKey, String responseBody,
                             String signatureToBeVerified) {
    if (StringUtil.isBlank(signatureToBeVerified)) return false;  // SIGNATURE_INVALID case
    String content = String.format("%s.%s.%s", clientId, responseTime, responseBody);
    java.security.Signature signature = java.security.Signature.getInstance("SHA256withRSA");
    PublicKey pubKey = KeyFactory.getInstance("RSA").generatePublic(
        new X509EncodedKeySpec(Base64.decodeBase64(tokapayPublicKey.getBytes("UTF-8"))));
    signature.initVerify(pubKey);
    signature.update(content.getBytes("UTF-8"));
    return signature.verify(Base64.decodeBase64(
        URLDecoder.decode(signatureToBeVerified, "UTF-8").getBytes("UTF-8")));
}
```

Key formats are portable: `PKCS8EncodedKeySpec` for private, `X509EncodedKeySpec` for public.

## 7. Common Pitfalls & Debugging (`20000004` / "signature invalid")

1. **Body mismatch.** The JSON signed must be byte-identical to the JSON sent. Frameworks that pretty-print, reorder keys, or re-escape Unicode break this.
2. **Wrong format for response verification.** Responses are **3 parts**, not 6.
3. **`Request-Time` precision.** Must be a 13-digit **millisecond** timestamp. Seconds (10 digits) won't verify.
4. **Wrong key direction.** Verify Toka responses with **Toka's public key**. Sign your own requests with **your private key**.
5. **Header parsing.** Comma-separated `key=value`. No spaces around `=`.
6. **URL-encoding.** The reference code URL-encodes the Base64 output before placing it in the header. URL-decode before Base64-decoding on receipt.
7. **Algorithm mismatch.** Default: `SHA256withRSA`.
8. **`keyVersion`.** Fixed at `1`. Rotations must be coordinated offline.

When in doubt: log the exact `Content_To_Be_Signed` string and resulting signature, and compare against a known-good fixture from Toka.
